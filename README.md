## Git rebase vs dev 브랜치 merge 차이

Git에서 **rebase**와 **merge**는 둘 다 브랜치를 통합하는 방법이지만, 히스토리를 다루는 방식이 완전히 다르다

---

## 1) 히스토리 형태

### **merge**

* 기존 커밋 히스토리를 **그대로 유지**하면서 양쪽 브랜치를 하나의 merge commit으로 합침.
* 히스토리가 실제 작업 순서를 충실하게 기록.
* 단, 히스토리가 복잡하고 분기점이 많아져서 보기 어려울 수 있음.

예시:

```
A - B - C (dev)
       \
        D - E (feature)
          \
           M (merge commit)
```

**팁:**
Merge는 협업 시 충돌 가능성을 줄이고, 누가 언제 무엇을 합쳤는지 명확히 보이는 장점이 있음.

---

### **rebase**

* 작업 브랜치(feature)를 dev 브랜치의 최신 상태 “위로” 올려서 **히스토리를 재작성**.
* 깔끔하고 직선적인 히스토리를 만들 수 있음.
* 대신 원래 커밋 ID가 바뀌므로, 이미 공유된 브랜치에서 하면 위험.

예시:

```
A - B - C (dev)
             \
              D' - E' (feature rebased)
```

**팁:**
Rebase는 “내 작업을 최신 기반 위에서 다시 쌓겠다”는 개념으로 이해하면 쉽다.
정리된 히스토리를 원할 때 로컬에서만 사용하면 안전함.

---

## 2) 충돌 처리 시점

### merge

* 충돌은 **merge 할 때 한 번** 발생.

### rebase

* 충돌은 **각 커밋을 옮길 때마다 발생**할 수 있음.

**팁:**
여러 커밋에서 충돌이 난다면 merge보다 rebase가 더 번거로울 수 있다.

---

## 3) 협업 관점

| 구분       | merge  | rebase         |
| -------- | ------ | -------------- |
| 히스토리     | 그대로 유지 | 재작성            |
| 위험성      | 낮음     | 공유 브랜치에서 사용 위험 |
| 히스토리 가독성 | 다소 복잡  | 매우 깔끔          |
| 충돌       | 한 번    | 여러 번 가능        |
| 협업 적합성   | O      | 공유 후에는 X       |

**팁:**
팀 규칙이 없다면 “공유 브랜치 → merge”, “개인 로컬 정리 → rebase”라는 기본 원칙을 따르면 대부분 안전하다.

---

## 4) 간단 요약

* **merge:** “있는 그대로 합치자.” 히스토리를 바꾸지 않음. 협업 안전.
* **rebase:** “최신 커밋 기준으로 내 작업을 다시 쓰자.” 히스토리가 깔끔. 단, 위험할 수 있음.
